# 指针不是魔法：地址、解引用与数组

> 系列：C 语言深度入门 · 第 3 篇  
> 前置知识：[第 2 篇：函数调用的秘密：栈帧与值传递](/posts/c-lang-stack-frame/)

---

上一篇结尾，我们写了一个 `swap` 函数，它没能真正交换两个数：

```c
void swap_fail(int a, int b) {
    int tmp = a;
    a = b;
    b = tmp;  // 改的是副本，x 和 y 纹丝不动
}
```

结论是：C 只有值传递，函数拿到的永远是拷贝。

那问题来了——如果真的需要让函数修改调用者的变量，该怎么办？

答案就是这一篇的主题：**指针（Pointer）**。

---

## 一、指针不是新东西，只是"存地址的变量"

初学者对指针望而生畏，多半是被"指针"这个名字吓到了，觉得它是某种神秘的底层机制。

其实它非常朴素：**指针就是一个变量，只不过它存的值是内存地址。**

我们已经知道，每个变量都有地址。`int x = 42` 声明后，`x` 住在内存的某个位置，比如 `0x7ffd...a4`。现在，把这个地址存到另一个变量里：

```c
int x = 42;
int *p = &x;   // p 存的是 x 的地址
```

这就是一个指针。`p` 的类型是 `int *`（指向 int 的指针），它的值是 `x` 的地址 `0x7ffd...a4`。

用图来看：

<img src="/images/c-lang-pointer-var.svg" alt="指针变量：p 存储 x 的地址，两者是独立的内存单元" style="max-width:100%; height:auto; display:block; margin:1.5rem auto;" />

注意：`p` 自己也住在内存里，也有自己的地址（`0x7ffd...98`）。它不是虚无缥缈的东西，它是一块 **8 字节**的内存（64 位系统上所有指针都是 8 字节），里面存着另一个变量的地址。

---

## 二、两个运算符：`&` 和 `*`

操作指针只需要掌握两个运算符：

- `&`（取地址）：从变量出发，得到它的地址
- `*`（解引用）：从地址出发，读取或写入那块内存的值

```c
int x = 42;
int *p = &x;   // & 取地址，p 现在存着 x 的地址

printf("%d\n", *p);   // * 解引用，通过 p 读取 x 的值 → 42

*p = 99;              // * 解引用 + 赋值，通过 p 修改 x 的值
printf("%d\n", x);    // x 已经被改了 → 99
```

`&` 和 `*` 是互为逆操作：

```
&x  →  得到 x 的地址（int *）
*p  →  通过地址 p 得到值（int）

*(&x)  == x   // 先取地址，再解引用，回到原地
&(*p)  == p   // 先解引用，再取地址，也回到原地
```

## 三、亲手验证指针的本质

代码说话：

```c
#include <stdio.h>

int main() {
    int x = 42;
    int *p = &x;

    printf("x  的值:   %d\n",   x);
    printf("x  的地址: %p\n",   (void*)&x);
    printf("p  的值:   %p\n",   (void*)p);    // p 存的就是 x 的地址
    printf("p  的地址: %p\n",   (void*)&p);   // p 自己也有地址
    printf("*p 的值:   %d\n",   *p);          // 解引用，等价于 x
    printf("指针大小:  %zu 字节\n", sizeof(p));

    *p = 99;
    printf("修改后 x = %d\n", x);

    return 0;
}
```

运行输出（地址数字因机器而异）：

```
x  的值:   42
x  的地址: 0x7ffd5a8c1c2c
p  的值:   0x7ffd5a8c1c2c   ← 和 x 的地址完全相同
p  的地址: 0x7ffd5a8c1c20   ← p 自己也有地址
*p 的值:   42
指针大小:  8 字节
修改后 x = 99
```

`p` 的值和 `&x` 完全相等——这就是"指针存地址"的直接证明。

## 四、用指针解决 swap 问题

现在回到上一篇留下的悬念。不传值，改传地址：

```c
#include <stdio.h>

void swap_ok(int *a, int *b) {
    int tmp = *a;  // 通过 a 读取 x 的值
    *a = *b;       // 通过 a 把 y 的值写进 x
    *b = tmp;      // 通过 b 把原 x 的值写进 y
}

int main() {
    int x = 1, y = 2;
    swap_ok(&x, &y);   // 传地址，不传值
    printf("x=%d, y=%d\n", x, y);  // x=2, y=1（成功！）
    return 0;
}
```

为什么成功了？看内存：

<img src="/images/c-lang-pointer-swap.svg" alt="指针传参：swap_ok 的 a、b 指向 main 的 x、y，解引用后修改原变量" style="max-width:100%; height:auto; display:block; margin:1.5rem auto;" />

`a` 和 `b` 不是 `x` 和 `y` 的副本，而是**通往 `x` 和 `y` 的路**。`*a = *b` 的意思是："去 `a` 指向的地方，把那里的值改成 `b` 指向的地方存的值。"

这就是指针传参的本质：**传地址，函数通过地址操作调用者的变量。**

## 五、数组名就是指针（差不多）

学完指针，来看数组。

你已经知道 `int arr[5]` 在内存里是 5 个 int 连续排列。有趣的是，数组名 `arr` 在大多数场景下会自动变成**首元素的地址**：

```c
int arr[5] = {10, 20, 30, 40, 50};

printf("%p\n", (void*)arr);        // 首元素地址
printf("%p\n", (void*)&arr[0]);    // 也是首元素地址，两者相等
```

两行输出的地址完全一样。这个行为叫**数组退化（Decay）**——在表达式中，数组名退化为 `&arr[0]`，类型变成 `int *`。

有了这一点，下标运算就有了新解释：

```c
arr[2]     ==  *(arr + 2)   // 从首地址偏移 2 个 int 后解引用
```

这不是比喻，是 C 语言标准的规定。所以这四种写法完全等价：

```c
arr[2]       // 熟悉的语法糖
*(arr + 2)   // 指针偏移 + 解引用
*(2 + arr)   // 加法可交换
2[arr]       // 合法但奇怪——不要这么写
```

相邻元素的地址差等于 `sizeof(int)` = 4 字节：

<img src="/images/c-lang-array-layout.svg" alt="数组内存布局：5个连续元素，arr 指向首元素，arr+2 指向第3个元素" style="max-width:100%; height:auto; display:block; margin:1.5rem auto;" />

## 六、退化的代价：长度信息消失

数组退化有个严重后果：**长度信息丢失**。

把数组传给函数时，函数收到的只是首地址，完全不知道数组有多长：

```c
void print_arr(int *arr, int len) {  // 必须另传 len！
    for (int i = 0; i < len; i++) {
        printf("%d ", arr[i]);
    }
}

int main() {
    int nums[5] = {1, 2, 3, 4, 5};
    print_arr(nums, 5);   // 必须手动传长度
}
```

函数参数里写 `int arr[]` 和 `int *arr` 是**完全一样的**，编译器都把它当指针。这也是为什么 C 代码里处理数组总要传一个额外的长度参数。

**没有边界检查是 C 的设计选择**——速度优先，安全靠程序员自律。越界访问不会报错，只会悄悄读写相邻内存：

```c
int arr[3] = {1, 2, 3};
arr[10] = 99;  // 编译通过，运行时悄悄覆盖了别的内存
```

这类 bug 极难调试，因为出错的地方和症状往往相距甚远。

## 七、C 字符串：有 `\0` 的 char 数组

C 没有内置的字符串类型，字符串就是**以 `\0`（ASCII 0）结尾的 `char` 数组**。

```c
char s[] = "hello";
// 等价于：
char s[] = {'h', 'e', 'l', 'l', 'o', '\0'};  // 6 个字节
```

`strlen`、`printf("%s")` 等函数都靠 `\0` 判断字符串在哪里结束。忘记 `\0` 的后果：

```c
char s[5];
s[0]='h'; s[1]='e'; s[2]='l'; s[3]='l'; s[4]='o';
// 没有 \0！strlen(s) 会一直往后读，直到碰到一个 0 字节，返回错误长度
```

还有一个容易混淆的区别：

```c
char s[] = "hello";   // 栈上的可写副本，sizeof = 6
char *p  = "hello";   // 指向只读代码段，sizeof = 8（指针大小）

s[0] = 'H';    // OK
p[0] = 'H';    // Segfault！只读内存不能写
```

`char s[]` 在栈上分配了 6 字节，`"hello"` 被复制进去，可以随意修改。  
`char *p` 只是一个指向代码段里字符串常量的指针，那块内存是只读的。

## 八、NULL 指针：约定的"无效地址"

最后一个重要概念：`NULL` 指针。

```c
int *p = NULL;   // p 不指向任何有效地址
```

`NULL` 的值是 0，操作系统不允许进程访问地址 0，所以解引用 `NULL` 指针会立即触发 Segfault：

```c
int *p = NULL;
*p = 42;   // 立即崩溃：Segmentation fault
```

这看起来很糟糕，但其实比悄悄写坏内存要好——**立即崩溃比悄悄出错更容易调试**。

正确做法是解引用前检查：

```c
if (p != NULL) {
    printf("%d\n", *p);
} else {
    printf("指针为空\n");
}
```

函数返回失败时，常用 `NULL` 表示"没有结果"（比如 `malloc` 失败返回 `NULL`）。这是 C 代码里随处可见的惯例。

## 小结

| 概念 | 本质 | 注意点 |
|------|------|--------|
| 指针 | 存地址的变量，8 字节（64位） | 解引用前必须确保有效 |
| `&` | 取变量地址 | 返回指针类型 |
| `*` | 解引用，读写地址处的值 | NULL 解引用 = Segfault |
| 数组名 | 大多数场景退化为首元素地址 | `sizeof`、`&arr` 不退化 |
| `arr[i]` | 等价于 `*(arr + i)` | 下标是指针运算的语法糖 |
| C 字符串 | `char` 数组 + `\0` 终止 | 忘记 `\0` = 越界读 |
| `char[]` vs `char *` | 可写副本 vs 只读常量指针 | 修改 `char *` 字面量 = Segfault |

## 动手练习

1. **打印指针关系**：声明 `int x = 100`，打印 `x`、`&x`、指针 `p` 的值、`&p`、`*p`、`sizeof(p)`。手动画出内存图。

2. **真正的 swap**：把上一篇的 `swap_fail` 改成 `swap_ok`，传地址而非值。打印调用前后的地址，确认 `a` 和 `b` 的地址与 `x`、`y` 相同（指针指向它们）。

3. **数组遍历**：声明 `int arr[5] = {10,20,30,40,50}`，用 `*(arr + i)` 遍历打印，同时打印每个元素的地址，验证相邻元素地址差恰好是 4 字节。

4. **越界实验**：声明 `int arr[3]` 和紧随其后的 `int x = 999`，写入 `arr[3]`，观察 `x` 是否被覆盖（用 AddressSanitizer：`gcc -fsanitize=address`）。

5. **字符串陷阱**：分别声明 `char s[] = "hello"` 和 `char *p = "hello"`，打印 `sizeof(s)` 和 `sizeof(p)`，尝试修改 `s[0]` 和 `p[0]`，观察区别。
